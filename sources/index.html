Posted on 2023-01-13

1. 如何去构建一个chat server

chat server要做的事情就是接收用户发来的信息，并且将这些信息广播给所有在聊天室中的用户。

  • 服务端处于空闲状态时，会一直监听给定的端口

  • 当用户发来一个连接请求时，服务端会启动一个新的goroutine去处理这个请求，处理的内容包括读取用户输入的信息和向用户发送信息

但是问题来了，如何使得一个用户发送的信息对聊天室所有的成员可见呢？

  • 设置一个map，里面记录了所有的用户连接信息

  • 设置一个broadcaster，使用select语句处理所有以下所有的channel，并且根据从channel中获取的信息，来对map进行修改

  • 设置一个channel entering，当用户发来一个连接请求时，会首先向entering中发送信息，此时broadcaster会将该用户的连接信息添加到map中
  • 设置一个channel leaving，当用户离开聊天室时，会向leaving中发送信息，此时broadcaster会将用户的连接信息从map中删除

  • 设置一个channel messages，当用户在聊天室中发送信息时，会向messages发送数据，此时broadcaster会遍历map，将从messages中获取到的信息，发送给聊天室中的所有成员

3. A more advanced chat server

项目地址：https://github.com/TutorialEdge/realtime-chat-go-react.git

这个项目用的不是我们之前的tcp socket了，而是建立在tcp之上的websocket通信协议，但是server设计的基本思想还是一样的。

websocket是在TCP上独立设计的全双工，有message概念的通信协议，为了兼容互联网现状，因此使用了和HTTP相同的端口和相似的协议形式，下面是一段websocket协议的报文：

由于websocket协议和HTTP协议相似，所以需要在websocket header中注明Upgrade和Connection字段。

当客户端和服务端建立起websocket连接时，由于协议是双向的，服务端可以不断地推送信息给客户端。

websocket解决了HTTP的被动性(只能由客户端向服务端发送HTTP请求，也就是说通信只能由客户端发起)和无状态(服务端不会记录来自客户端的信息，每一次客户端发送HTTP请求时，都要携带之前的信息)的问题。

如果不使用websocket协议，那么则需要使用ajax或者long poll来实现双方的通信。

ajax是客户端不断地发送HTTP请求询问服务端是否有新的信息推送给客户端，没有则返回。这对服务端处理HTTP请求的速度有很高的要求。

而long poll采用的是阻塞模型，如果客户端询问服务端，但是服务端没有新的信息推送给客户端时，线程会阻塞，直到服务端有信息，并将信息推送给客户端。这对服务端并发能力提出了很高的要求。

8.1.3 unidirectional channel types

当一个函数接受一个channel作为参数时，在许多场景下，channel只用来接收或者只用来发送。go语言提供了单方向的channel类型，chan<- int表示一个只发送int的channel，<-chan int表示一个只接收int的channel，如果向一个只接收的channel中发送数据，或者从只发送的channel接收数据，那么在编译阶段就不能通过。

只有发送者所在的goroutine才可以调用close函数，对只接收的channel调用close，会触发编译错误。

8.2 select

当一个goroutine需要从两个channel接收信号和数据时，如果没有goroutine向第一个channel发送信号或者数据时，该goroutine便会阻塞，从而也无法从第二个channel中接收到信号和数据，即使有其他goroutine向第二个channel中发送数据。

此时可以利用select语句实现多路复用multiplex。下面是select语句的一般形式，和switch语句非常相似：

Each case specifies a communication (a send or receive operation on some channel) and an associated block of statements. A receive expression may appear on its own, as in the first case, or within a short variable declarat ion, as in the second case; the second form lets you refer to the received value.

select会等待case中有能够执行的case时去执行。当条件满足时，select才会去通信并执行case之后的语句；这时候其它通信是不会执行的。一个没有任何case的select语句写作select{}，会永远地等待下去。

考虑下面一段程序：

7.3 type assertions && type switches

类型断言是对interface value一种操作，语法形式类似x.(T)，其中x是interface value，T是asserted type.根据T的不同类型，会出现以下两种情况：

T是一个具体类型

当T是具体类型是，类型断言检查x的dynamic type是否和T是相同的；如果相同，那么类型断言返回的是x的dynamic value；如果不同，那么类型断言会失败，触发panic：

T是一个接口类型

当T是一个接口类型时，类型断言会检查x的dynamic type是否能满足T，如果能，类型断言返回一个interface value，它的dynamic type变成了T，但是dynamic value保持不变；如果不能，那么类型断言失败，触发panic：

通常来说，我们并不确定一个interface value的类型，我们也不想在判断过程中触发panic。

If the type assertion appears in an assignment in which two results are expected, such as the following declarations, the operation does not panic on failure but instead returns an additional second result, a boolean indicating success.

在书中，作者介绍了两者使用类型断言的场景：

  • 基于类型断言识别错误类型

    有时候我们需要判断错误的类型，如os包中的IsExist等函数。判断错误类型的一种方式是对Error方法返回的字符串进行字符串匹配，但是这样会显得不够robust，因此在go的os包中，为不同的错误定义了错误类型，如PathError是用来描述在文件路径操作中涉及到的失败，可以利用类型断言判定得到的error是否是PathError类型的，来实现IsNotExist函数

  • 基于类型断言查询接口

    有一些接口io.Writer的方法如Write接受的参数是[]byte类型，但是我们通常传入的是字符串，需要进行类型转换，而从string到[]byte的过程会发生拷贝，会消耗一部分内存。通常情况下，实现io.Writer接口的类型通常也会实现WriteString方法，该方法直接接受string类型参数。当要传递给Write函数一个string参数时，在强制类型转换之前，可以先用类型断言判定 interface value 是否已经实现WriteString方法，如果是，那么可以直接调用该方法，不必再进行额外的强制类型转换了。

5.3 panic && recover

go的panic机制类似于其他语言的异常机制，但是panic会引起程序的崩溃，因此panic一般用于严重的错误。对于大部分的错误和漏洞，应该使用go提供的错误机制。

当panic异常发生时，程序会中断运行，执行当前goroutine中被defer延迟的函数，随后程序输出日志信息，包括panic value和堆栈跟踪信息。

通常来说，不应该对panic做任何处理，但是有时我们可以从异常中恢复，或者说在程序崩溃前，做一些操作，如web服务器在运行时出现panic异常，应该在崩溃前把所有的连接都关闭，否则客户端会一直处于等待状态。

4.2 sha256算法

一个n位的哈希函数就是一个从任意长的消息到n位哈希值的映射，一个n位的加密哈希函数就是一个单向的、避免碰撞的n位哈希函数。这样的函数是目前在数字签名和密码保护当中极为重要的手段。当前比较流行的哈希函数主要有128位的MD4和MD5和160位的SHA-1，SHA-2族有着更多位的输出哈希值，破解难度更大，能够提高更高的安全性。

对于任意长度的消息，SHA256都会产生一个256位的哈希值，称作消息摘要。这个摘要相当于是个长度为32个字节的数组，通常有一个长度为64的十六进制字符串来表示，其中1个字节=8位，一个十六进制的字符的长度为4位。

计算"Blockchain"经过SHA256得到的哈希值：

输出：

625da44e4eaf58d61cf048d168aa6f5e492dea166d8bb54ec06c30de07db57e1

总体上，HSA256与MD4、MD5以及HSA-1等哈希函数的操作流程类似，待哈希的消息在继续哈希计算之前首先要进行以下两个步骤：

  • 对消息进行补位处理，是的最终的长度是512位的倍数
  • 然后以512位为单位对消息进行分块为\(M^{(1)}, M^{(2)}, ..., M^{(N)}\)

将消息进行逐个处理：从一个固定的初始哈希\(H^{(0)}\)开始，进行以下序列的计算： \[H^{(i)} = H^{(i-1)}+C_{M^{(i)}}(H^{(i-1)})\] 其中\(C\)是SHA256的压缩函数，\(+\)是\(\rm mod \;2^{32}\)加法，\(H^{(N)}\)是消息区块的哈希值
