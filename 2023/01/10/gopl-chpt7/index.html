7.2 interface value

7.2.1 interface value

Conceptually, a value of an interface type, or interface value, has two components, a concrete type and a value of that type. These are called the interface’s dynamic type and dynamic value.

开始时声明的变量w，类型为nil，值也为nil。如果此时调用接口的Write方法，会导致panic：

考虑下面一段程序：

如果debug被改成了false，那么函数f中的防御性措施if out != nil将不会生效。这是因为向函数f传入的值buf的类型是*bytes.Buffer，也就是说，将buf传给f后，在f中的局部变量out的dynamic type始终是*bytes.Buffer，尽管它的dynamic value为nil，因此 interface value out始终不可能为nil，防御性措施无效，仍然会触发panic。

正确的做法是将main函数中var buf *bytes.Buffer 改成var buf io.Writer，这样在未初始化的前提下，buf的dynamic type和dynamic value便都是nil，out == nil成立。

7.3 type assertions && type switches

类型断言是对interface value一种操作，语法形式类似x.(T)，其中x是interface value，T是asserted type.根据T的不同类型，会出现以下两种情况：

T是一个具体类型

当T是具体类型是，类型断言检查x的dynamic type是否和T是相同的；如果相同，那么类型断言返回的是x的dynamic value；如果不同，那么类型断言会失败，触发panic：

T是一个接口类型

当T是一个接口类型时，类型断言会检查x的dynamic type是否能满足T，如果能，类型断言返回一个interface value，它的dynamic type变成了T，但是dynamic value保持不变；如果不能，那么类型断言失败，触发panic：

通常来说，我们并不确定一个interface value的类型，我们也不想在判断过程中触发panic。

If the type assertion appears in an assignment in which two results are expected, such as the following declarations, the operation does not panic on failure but instead returns an additional second result, a boolean indicating success.

可以利用 type switch来对该过程进行简化：

在书中，作者介绍了两者使用类型断言的场景：

  • 基于类型断言识别错误类型

    有时候我们需要判断错误的类型，如os包中的IsExist等函数。判断错误类型的一种方式是对Error方法返回的字符串进行字符串匹配，但是这样会显得不够robust，因此在go的os包中，为不同的错误定义了错误类型，如PathError是用来描述在文件路径操作中涉及到的失败，可以利用类型断言判定得到的error是否是PathError类型的，来实现IsNotExist函数

  • 基于类型断言查询接口

    有一些接口io.Writer的方法如Write接受的参数是[]byte类型，但是我们通常传入的是字符串，需要进行类型转换，而从string到[]byte的过程会发生拷贝，会消耗一部分内存。通常情况下，实现io.Writer接口的类型通常也会实现WriteString方法，该方法直接接受string类型参数。当要传递给Write函数一个string参数时，在强制类型转换之前，可以先用类型断言判定 interface value 是否已经实现WriteString方法，如果是，那么可以直接调用该方法，不必再进行额外的强制类型转换了。
